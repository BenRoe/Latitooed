# Phase 4 - UI Design Contract

> Visual and interaction contract for batch progress, per-file result diagnostics, best-effort video result behavior, recent coordinates, and compact batch history. Generated by gsd-ui-researcher for checker review.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | Manual native SwiftUI design system |
| Preset | Not applicable |
| Component library | Native SwiftUI controls only |
| Icon library | SF Symbols |
| Font | System font through SwiftUI text styles |

Implementation must preserve the existing dense macOS utility layout built in Phases 1 and 2. Use SwiftUI, MapKit, SwiftData, and existing `AppDesign` constants. Do not introduce third-party UI frameworks, web component registries, UIKit/AppKit color bridging in SwiftUI views, fixed font sizes, a landing page, marketing copy, or decorative/card-heavy redesign.

No shadcn gate applies because this is a native macOS SwiftUI project, not React, Next.js, or Vite.

---

## Layout Contract

| Region | Contract |
|--------|----------|
| Main window | Preserve the existing `HSplitView`. File intake, selected-file results, detail, and history stay on the left; coordinate controls and map stay on the right. |
| Selected-files table | Remains the primary batch result surface. Rows update only after each file completes. Do not add a separate results drawer, report view, or modal result summary. |
| Bottom-left detail/result area | Continue using the existing detail panel below the table. It shows the selected file's result message and warning/failure diagnostics. |
| Batch history | Add a compact recent batch history section below the selected-file detail/result area in the left column. It must not push the table below a usable height. |
| Footer/status area | Existing footer shows active write progress and completed batch summary. Progress must be filename-first. |
| Coordinate panel | Recent coordinates appear near the existing search/manual controls, above the map and below or adjacent to search/manual entry. The map remains the dominant right-panel surface. |
| Cancellation | Do not add active cancellation UI, a Cancel command, cancelled row state, or future cancellation deferral copy. |

---

## Visual Hierarchy

| Priority | Element | Rule |
|----------|---------|------|
| 1 | Selected-files table | Main operational surface during and after a batch. Display name remains the first scan target, followed by type, GPS, and latest result. |
| 2 | Footer progress | During an active write, progress copy is the clearest batch status and sits beside the existing Apply Location command. |
| 3 | Selected-file detail diagnostics | Secondary review surface for the selected warning/failure row. Keep diagnostics collapsed by default. |
| 4 | Recent coordinates | Compact reuse affordance near coordinate controls. It must not compete with search/manual entry. |
| 5 | Recent batch history | Compact informational history below details. Counts-only rows; not a restore surface. |

Success rows stay visually quiet. Warning and failure rows may use semantic warning/destructive styling plus iconography, but color must not be the only signal.

## Spacing Scale

Use the existing `AppDesign.Spacing` scale. Declared values are multiples of 4:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, disclosure chevron gaps, badge internal gaps |
| sm | 8px | Result row cell gaps, recent-coordinate row gaps, history row spacing |
| md | 16px | Table-to-detail spacing, detail-to-history spacing, coordinate control groups |
| lg | 24px | Existing left/right column outer padding and footer horizontal padding |
| xl | 32px | Reserved for large empty-state breathing room; avoid in Phase 4 additions |
| 2xl | 48px | Not used in Phase 4 additions |
| 3xl | 64px | Not used in Phase 4 additions |

Exceptions: macOS table row heights, SwiftUI controls, `ProgressView`, and disclosure controls may use system metrics. Interactive controls must preserve at least the existing 44px activation area when presented as compact rows or icon-backed controls.

## Typography

Use the Phase 1/2 hierarchy and SwiftUI text styles. Do not add fixed font sizes in implementation.

| Role | Size | Weight | Line Height | SwiftUI Style |
|------|------|--------|-------------|---------------|
| Label | 12px | Regular 400 | 1.4 | `.caption` or equivalent platform label style |
| Body | 14px | Regular 400 | 1.5 | `.body` |
| Heading | 17px | Bold 700 | 1.25 | `.headline` with `bold()` only when needed |
| Display | 24px | Bold 700 | 1.2 | Not used in Phase 4 additions |

Footer progress, history metadata, recent-coordinate rows, and diagnostic disclosure labels use label/body styles. Do not use `.caption2`, negative letter spacing, or extra weights such as medium/semibold.

## Color

Use adaptive native colors. Hex values are light-mode references only, not fixed implementation colors.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `appBackground` reference `#F5F5F5` | Window background and broad split layout surfaces |
| Secondary (30%) | `contentSurface` reference `#FFFFFF` | Table surface, detail panel, history section, coordinate control surfaces |
| Accent (10%) | System accent color, blue reference `#0A84FF` | Apply Location primary CTA, selected row focus, selected recent coordinate, selected history coordinate reuse, keyboard focus ring |
| Warning | `warning` reference `#B26A00` | Warning result rows, warning diagnostic disclosure, best-effort video warning messages |
| Destructive | `destructive` reference `#D70015` | Failure result rows and existing overwrite confirmation only |

Accent reserved for: Apply Location, active table/row selection focus, selected recent-coordinate reuse row, selected batch-history coordinate reuse, and keyboard focus rings. Do not use accent for all result badges, all history rows, success states, diagnostics, or decorative section borders.

## Component Contract

| Component | Contract |
|-----------|----------|
| Footer progress | While writing, replace generic status text with `Writing {filename} ({index} of {total})`. The filename comes first and may truncate with full text available through help/tooltip. |
| Footer completed summary | After completion, show counts-only summary such as `9 updated, 2 warnings, 1 failed.` Use the existing notice/status area; do not add a completion modal. |
| Apply Location button | Keep existing `Button("Apply Location", systemImage: "location.fill", action: ...)`. Disable while a batch is running. Do not add Cancel. |
| Latest Result column | Keep the existing column. Success is quiet; warnings and failures use distinct status labels/icons. MOV and MP4 success uses the same success style as images. |
| Selected-file detail | Show selected filename, folder, latest result, latest user-facing message, and optional collapsed diagnostics for warning/failure rows only. |
| Diagnostics disclosure | Use a compact native disclosure such as `DisclosureGroup("Diagnostics")` only when diagnostic detail exists and latest result is warning or failure. Diagnostics are text-selectable. |
| Recent coordinates | Compact list/section near coordinate search/manual controls. Rows show place label when available, otherwise formatted coordinates at 6 decimals. |
| Recent batch history | Compact section below selected-file detail/result area. Each row shows timestamp, coordinate label, total file count, and success/warning/failure counts. |
| History coordinate reuse | A history row may offer coordinate reuse through a clearly labeled row/button. Reuse updates the active coordinate panel only. |
| History privacy boundary | History rows must not show previous filenames, file paths, per-file result details, bookmarks, thumbnails, or restore affordances. |

## Interaction Contract

| Interaction | Contract |
|-------------|----------|
| Active batch progress | Before each awaited write, publish current filename, 1-based index, and total count. Rows change only after the file completes. |
| Batch running state | Disable Apply Location and other write-start paths while running. Do not expose user cancellation even if internal task cancellation remains available for infrastructure. |
| Row selection | Selecting a row updates the bottom-left detail panel with that file's latest message and any allowed diagnostics. |
| Success result review | Success rows show `GPS metadata updated.` or equivalent app-owned success copy. No diagnostic disclosure appears for success rows. |
| Warning/failure review | Warning/failure rows show user-facing message in the detail panel. If diagnostics exist, reveal them only after the user expands the disclosure. |
| Video result behavior | MOV and MP4 share user-facing behavior with images on clean ExifTool exit. Show warnings or failures only when the helper reports them. |
| Recent coordinate selection | Activating a recent coordinate updates the same selected-coordinate state used by Apply Location and syncs manual fields/map target. |
| Batch history selection | Activating a batch history entry may reuse that coordinate, but must not restore prior files, prior selected rows, or prior per-file results. |
| Empty recent coordinates | Show quiet inline empty copy near coordinate controls. Do not show a modal or large empty state. |
| Empty batch history | Show quiet inline empty copy below the detail panel. Do not imply files can be recovered later. |

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Apply Location |
| Active footer progress | Writing {filename} ({index} of {total}) |
| Completed footer summary | {updatedCount} updated, {warningCount} warnings, {failureCount} failed. |
| Single warning summary grammar | {updatedCount} updated, 1 warning, {failureCount} failed. |
| Video success message | GPS metadata updated. |
| Warning diagnostics disclosure | Diagnostics |
| Diagnostics empty state | No diagnostic details were reported. |
| Recent coordinates heading | Recent Coordinates |
| Empty recent coordinates | Recent coordinates will appear after you apply a location. |
| Batch history heading | Recent Batches |
| Empty batch history | Recent batch summaries will appear after a write completes. |
| Batch history row | {timestamp} - {coordinateLabel} - {fileCount} files - {updatedCount} updated, {warningCount} warnings, {failureCount} failed. |
| History reuse action | Use Coordinate |
| History privacy guard copy | Do not add visible copy promising restore, reopen, or reconnect behavior for previous files. |
| Error state | Metadata could not be written. Select the row for details. |
| Destructive confirmation | Existing overwrite confirmation only: GPS metadata will be overwritten in the selected files. The original metadata cannot be restored through this app. |

Do not use `Cancel`, `Cancelled`, `Restore Files`, `Reopen Batch`, `View Previous Files`, or any copy implying previous media or per-file results can be restored from SwiftData history.

## Persistence And Privacy Contract

| Area | Contract |
|------|----------|
| Recent coordinates | Persist compact coordinate records only: label, latitude, longitude, and last-used timestamp. Labels use selected place name when available, otherwise formatted coordinates. |
| Batch history | Persist compact summary records only: timestamp, coordinate label/value, total file count, success count, warning count, and failure count. |
| Forbidden persisted data | Do not persist media contents, file URLs, bookmark data, thumbnails, display-name lists, per-file result rows, diagnostic logs, or enough information to reconnect old files. |
| Save behavior | Correctness-sensitive inserts/updates/pruning must call explicit `ModelContext.save()`. |
| Restore behavior | History is informational plus coordinate reuse only. It is never a prior-file restore surface. |

## Accessibility Contract

| Area | Requirement |
|------|-------------|
| Text | Use Dynamic Type-compatible SwiftUI text styles. Do not force fixed font sizes. |
| Footer progress | Progress text must be accessible as text, not only a visual progress bar. |
| Result states | Warning/failure/success must include text and icons; color alone is insufficient. |
| Diagnostics | Disclosure label must be keyboard reachable and announce expanded/collapsed state. Diagnostic text must be selectable and readable by VoiceOver. |
| Recent coordinate/history rows | Use labeled `Button` or row controls with accessible names such as `Use Coordinate, Berlin`. |
| Long filenames | Truncate in constrained rows, but expose full filename through row detail and/or help text. |
| Motion | Any progress or row update transitions must be subtle and respect Reduce Motion. |
