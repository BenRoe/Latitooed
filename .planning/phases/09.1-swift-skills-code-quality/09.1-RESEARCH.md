#### DATA-MEDIUM: #Unique on RecentCoordinate + upsert refactor
[VERIFIED: ctx7 /websites/developer_apple_swiftdata — #Unique<T>([PartialKeyPath]...) confirmed;
placed inside @Model body]

Add to `RecentCoordinate.swift`:
```swift
@Model final class RecentCoordinate {
    #Unique<RecentCoordinate>([\.latitude, \.longitude])
    // ... existing fields
}
```

With `#Unique` in place, SwiftData enforces uniqueness at the store level. The upsert pattern:
insert + set all fields. If a row with the same (latitude, longitude) already exists, SwiftData
merges the insert into an update of the existing record.

Simplify `BatchHistoryStore.recordRecentCoordinate`:
```swift

    let coord = RecentCoordinate(
        label: label, latitude: coordinate.latitude,
        longitude: coordinate.longitude, lastUsedAt: lastUsedAt
    )
    modelContext.insert(coord)
    // Update mutable fields (SwiftData merges on unique key)
    coord.label = label
    coord.lastUsedAt = lastUsedAt
    try pruneRecentCoordinates()
    try modelContext.save()
}
```

Remove `existingRecentCoordinate(for:)` private method (no longer needed).

has no CloudKit (confirmed: STATE.md, no `CloudKit` entitlement). Safe to add.

### #Unique upsert pattern

```swift
// Source: ctx7 /websites/developer_apple_swiftdata
@Model final class Item {
    #Unique<Item>([\.uniqueKey])
    var uniqueKey: String
    var mutableField: String
}
// Usage: insert sets uniqueKey; SwiftData merges on conflict
func upsert(key: String, value: String, in context: ModelContext) {
    let item = Item(uniqueKey: key, mutableField: value)
    context.insert(item)
    // If uniqueKey already existed, SwiftData updates the existing record.
    // Set mutable fields after insert to ensure the merge carries new values.
    item.mutableField = value
}
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead |
|---------|-------------|-------------|
| Schema uniqueness enforcement | Manual O(n) scan + fetch | `#Unique` macro — SwiftData enforces at store level |
| Schema migration safety | Implicit lightweight migration | `VersionedSchema` + `SchemaMigrationPlan` — gives explicit rollback path |
| Empty-state UI | Custom Text + frame | `ContentUnavailableView` — matches HIG, free accessibility |
| Geometry measurement | `GeometryReader` wrapper | `onGeometryChange(for:of:action:)` — no layout pass overhead |
| Source location in helpers | Manual 4-param decomposition | `#_sourceLocation` macro — single parameter, correct default |

## Common Pitfalls

### Pitfall 1: #Unique + upsert — mutable fields not updated on merge
**What goes wrong:** Insert a duplicate; SwiftData merges to existing record but mutable
fields retain old values if you don't set them after `insert`.
**How to avoid:** Always set all mutable fields after `modelContext.insert(obj)`.

### Pitfall 2: AppDesign.Radius rename — partial update leaves build broken
**What goes wrong:** Renaming the token but missing one call site causes a build error that
blocks the test run.
**How to avoid:** Use the grep inventory in this document (14 sites, 9 files). Run a build
after each file update, or update all files in one commit.

### Pitfall 3: waitUntil consolidation — import / visibility
**What goes wrong:** Moving `waitUntil` to a shared file requires it to be a `private`
free function in the test target. Each test file has its own private copy today; keeping
them private but removing the duplicate means one file loses its copy.
**How to avoid:** Define one copy as a module-level `internal` free function in a new
`TestHelpers.swift` file in the test target, or keep it `private` in one file and add
`fileprivate` access from the other. Simplest: one `private` copy per file (both are
already `private`), but unify the implementation (yield, not sleep).

### Pitfall 4: VersionedSchema — do not forget to update .modelContainer call
**What goes wrong:** Defining `SchemaV1` and `AppMigrationPlan` but not wiring them into
`.modelContainer(migrationPlan:for:)` means the schema change has no effect.
**How to avoid:** The GPSMetadataEditorApp.swift change is the load-bearing step.

### Pitfall 5: deinit on @Observable @MainActor class
**What goes wrong:** `@Observable` classes are ARC-managed; `deinit` runs off the main
actor when the refcount drops to zero. Calling `Task.cancel()` (a non-isolated, non-async
method) from `deinit` is safe — it just sets a cooperative flag.
**How to avoid:** Keep the cancel call simple (`activeGPSScanTask?.cancel()`); do not await
anything in `deinit`.
