# Phase 09.1: Code Review Report

**Reviewed:** 2026-05-28
**Depth:** standard
**Files Reviewed:** 21
**Status:** issues_found

## Summary

Phase 09.1 applied Swift Concurrency fixes, SwiftData VersionedSchema + migration plan, Swift Testing improvements, and SwiftUI accessibility/API modernisation. The majority of changes are well-reasoned and the concurrency architecture is sound. Three blockers exist: a `waitUntil` polling helper that is structurally unsound under cooperative scheduling (no Task.sleep between yields), a SwiftData schema wiring gap where `@Model` types are not nested inside the `VersionedSchema` enum (defeating migration correctness), and a task-cancellation hole in the GPS scan path that lets a stale scan overwrite results from a newer one on the `@MainActor`. Six warnings cover lesser correctness risks and maintainability issues.

## Critical Issues

### CR-01: `waitUntil` polling loop is not guaranteed to observe async side-effects

**File:** `GPSMetadataEditorTests/FileIntakeViewModelTests.swift:413-422`, `GPSMetadataEditorTests/MetadataBatchViewModelTests.swift:304-313`

**Issue:** `waitUntil` calls `Task.yield()` up to 20 times and returns an `Issue.record` failure (not a thrown error) if the condition is never met. Two separate problems compound:

1. `Task.yield()` surrenders the current task's turn but does **not** provide a minimum delay. Under the cooperative thread pool it may return immediately when there is no other work queued, meaning 20 yields can elapse before the background Task producing the condition has had any real opportunity to run on the executor. The `intakeCommandScansAcceptedFilesForExistingGPSCoordinates` test in `FileIntakeViewModelTests` chains a GPS-scan Task that calls an `async` fake reader; 20 cooperative yields are not a reliable upper bound for that async work to land back on the main actor.

2. The helper records an `Issue` instead of throwing, so test execution continues past the failing assertion. Subsequent `#expect` calls then read stale state and produce misleading secondary failures rather than stopping cleanly.

**Fix:**
```swift
private func waitUntil(
    _ condition: @MainActor @escaping () -> Bool,
    timeout: Duration = .milliseconds(500),
    sourceLocation: SourceLocation = #_sourceLocation
) async throws {
    let deadline = ContinuousClock.now + timeout
    while ContinuousClock.now < deadline {
        if await condition() { return }
        try await Task.sleep(for: .milliseconds(10))
    }
    Issue.record("Condition not met within \(timeout)", sourceLocation: sourceLocation)
}
```
A small fixed sleep (10 ms) between polls gives the cooperative scheduler a genuine opportunity to run other tasks, and a real deadline prevents infinite hangs without relying on a magic iteration count.

### CR-02: SwiftData `@Model` types are not nested inside `SchemaV1` — migration correctness is broken

**File:** `GPSMetadataEditor/GPSMetadataEditorApp.swift:4-12`, `GPSMetadataEditor/Features/BatchHistory/Models/RecentCoordinate.swift`, `GPSMetadataEditor/Features/BatchHistory/Models/BatchRunSummary.swift`

**Issue:** `VersionedSchema` requires that each versioned model type live **inside** the corresponding schema enum so SwiftData can snapshot the exact shape of the schema at that version. The current code declares `SchemaV1` with `models: [RecentCoordinate.self, BatchRunSummary.self]`, but those classes are top-level types, not `SchemaV1.RecentCoordinate` and `SchemaV1.BatchRunSummary`. When a second schema version is introduced and a lightweight migration stage maps `SchemaV1.RecentCoordinate` → `SchemaV2.RecentCoordinate`, SwiftData cannot distinguish the two because both refer to the same top-level class. This causes the migration plan to have no useful effect and means any future migration will either be skipped silently or trap at runtime.

The `AppMigrationPlan.schemas` list and `stages: []` are correct for a single-version app, but the structural mistake will create an irreversible data-model bind when a second version is needed.

**Fix:** Nest the model type declarations inside the schema enum (standard SwiftData VersionedSchema pattern):

```swift
enum SchemaV1: VersionedSchema {
    static var versionIdentifier: Schema.Version { .init(1, 0, 0) }
    static var models: [any PersistentModel.Type] { [RecentCoordinate.self, BatchRunSummary.self] }

    @Model final class RecentCoordinate { /* ... */ }
    @Model final class BatchRunSummary { /* ... */ }
}
```

Then reference `SchemaV1.RecentCoordinate` everywhere, and alias the latest version for app code with `typealias RecentCoordinate = SchemaV1.RecentCoordinate`. The `modelContainer` call should reference the top-level alias, not the schema-nested type directly, to keep call sites clean.

### CR-03: GPS scan task cancellation does not prevent stale results from landing

**File:** `GPSMetadataEditor/Features/FileIntake/FileIntakeViewModel.swift:162-171`, `357-385`

**Issue:** `intake(urls:source:)` cancels the previous `activeGPSScanTask` and starts a new one, which is correct. However, `refreshGPSStatuses(for:)` does not check `Task.isCancelled` between files. After cancellation is signalled the loop continues to call `gpsMetadataReader.gpsStatus(for:)` for every remaining file. Each result is then written via `replaceGPSStatus` — which guards against overwriting `.updated` status but not against overwriting results that belong to a different intake batch.

Concretely: user drops 50 files (scan A starts), immediately drops 50 different files (scan A cancelled, scan B starts). Scan A's `gpsStatus` calls have already been launched and may complete after scan B has populated the new `selectedFiles` array. `replaceGPSStatus` matches on `file.id`, so it will silently no-op for files not in the current list — but only if the IDs differ. If the same URL is re-added across batches the file gets a new `SelectedMediaFile` with a fresh `UUID`, so the no-op protection works by accident. The real risk is CPU waste for large batches; correctness is mostly protected. But the cancellation propagation gap is still a structural defect that will matter the moment the reader does meaningful I/O.

**Fix:** Add a cooperative cancellation check at the top of the loop body:

```swift
private func refreshGPSStatuses(for files: [SelectedMediaFile]) async {
    for file in files {
        try? Task.checkCancellation()  // exit loop if task was cancelled
        guard !Task.isCancelled else { return }
        guard let gpsStatus = await gpsMetadataReader.gpsStatus(for: file) else {
            continue
        }
        replaceGPSStatus(gpsStatus, for: file)
    }
}
```

Alternatively, change the return type to `async throws` and propagate the `CancellationError` to the task boundary, letting Swift Task machinery handle the exit cleanly.

## Warnings

### WR-01: `onGeometryChange` closure captures mutable state from wrong scope

**File:** `GPSMetadataEditor/Features/CoordinateSelection/Views/CoordinateSearchPanel.swift:60`

**Issue:**
```swift
.onGeometryChange(for: CGFloat.self) { $0.size.height } action: { fieldHeight = $0 }
```
`onGeometryChange(for:of:action:)` was introduced in iOS 17 / macOS 14. This is a macOS app; the deployment target is not visible in the reviewed files but if any device running macOS 13 is in scope this call will fail to compile (or silently be unavailable). More importantly, the modifier is applied to the `HStack` at line 38 but the `fieldHeight` state it feeds is used only to offset the dropdown overlay at line 65. If the `HStack` is replaced or the view hierarchy changes, the geometry reading silently stops updating without any compiler warning.

This is a `WARNING` rather than a `BLOCKER` because the logic is correct on macOS 14+, but it is fragile: the two-clause API separates the transform closure from the action closure syntactically, making it easy to swap the arguments and produce a hard-to-debug geometry bug. A comment anchoring the intent would reduce future risk.

**Fix:** Add a comment stating the minimum OS requirement and what geometry is being captured:
```swift
// Captures the search field's rendered height so the dropdown can be pinned
// exactly below it. Requires macOS 14+ (onGeometryChange API).
.onGeometryChange(for: CGFloat.self) { $0.size.height } action: { fieldHeight = $0 }
```

### WR-02: `TableSelectionNormalizer` `Task { @MainActor in await Task.yield() }` races with SwiftUI state

**File:** `GPSMetadataEditor/Features/FileIntake/Views/SelectedFilesTable.swift:119-122`

**Issue:**
```swift
Task { @MainActor in
    await Task.yield()
    selection.wrappedValue = [selectedID]
}
```
This pattern posts a one-frame-delayed mutation to override the `Table`'s own selection handling. The yield is intentional (let the table process its own click first), but the binding write happens unconditionally after the yield even if:

- The user has already moved focus to a different row before the yield resumes.
- The `files` array has changed under the task (e.g. a new intake completed during the yield).

`selectedID` is captured from `files[row].id` at mouse-down time, so if `files` shrinks and the captured ID no longer exists, `selection.wrappedValue = [selectedID]` silently inserts a phantom ID into the set. The view model's `selectedFileReview` then returns `.none` (because `selectedFileIDs.contains(id)` finds no matching file), which is harmless but confusing.

Additionally, the global `NSEvent.addLocalMonitorForEvents(matching: .leftMouseDown)` is process-wide. If multiple `SelectedFilesTable` instances existed simultaneously, each would install its own monitor and all would fire on every click.

**Fix:** Guard the binding write with an existence check:
```swift
Task { @MainActor in
    await Task.yield()
    guard files.contains(where: { $0.id == selectedID }) else { return }
    selection.wrappedValue = [selectedID]
}
```

### WR-03: `applyMetadata` does not check `Task.isCancelled` between files; long batch cannot be interrupted

**File:** `GPSMetadataEditor/Features/FileIntake/FileIntakeViewModel.swift:258-291`

**Issue:** The metadata batch loop iterates all `selectedFiles` sequentially without any cancellation check. If the user closes the window or the app enters background mid-batch, the task cannot be interrupted between `writer.writeGPS` calls. For large file sets this means the app cannot respond to lifecycle events until the entire batch finishes.

There is also a subtle ordering issue: `isMetadataBatchRunning = true` is set synchronously, but `resetMetadataWriteResults()` runs synchronously too — both before the first `await`. If `applyMetadata` is called from a non-isolated context (not currently possible, but the function is not isolated to `@MainActor` in its signature — the isolation comes from the enclosing `@Observable @MainActor` class), a re-entrancy window exists between the two state mutations.

**Fix:** Add cancellation checks inside the loop:
```swift
for (index, file) in files.enumerated() {
    try Task.checkCancellation()
    // ... rest of loop
}
```
Change return type to `async throws` to propagate cancellation, and update `applyMetadataIfConfirmed` accordingly.

### WR-04: `BatchHistoryStore.recordBatchRun` double-saves — `recordRecentCoordinate` calls `save()` then the method calls `save()` again

**File:** `GPSMetadataEditor/Features/BatchHistory/Services/BatchHistoryStore.swift:68-91`

**Issue:** `recordBatchRun` calls `recordRecentCoordinate(...)` at line 75, which calls `modelContext.save()` at line 50. Then `recordBatchRun` calls `modelContext.save()` again at line 90. Between the two saves, `BatchRunSummary` has been inserted but not yet saved, so the first `save()` commits an incomplete transaction. If the app crashes between the two saves (e.g. because `pruneBatchRunSummaries` throws), the `RecentCoordinate` is persisted but the `BatchRunSummary` is lost, leaving the two tables in an inconsistent state.

**Fix:** Extract a private `_recordRecentCoordinate` that does not save, call it from `recordRecentCoordinate` (which adds the save), and call the private variant from `recordBatchRun` so all mutations are committed in a single `save()`:

```swift
private func insertRecentCoordinate(label: String, coordinate: CoordinateSelection, lastUsedAt: Date) throws {
    let coord = RecentCoordinate(...)
    modelContext.insert(coord)
    coord.label = label
    coord.lastUsedAt = lastUsedAt
    try pruneRecentCoordinates()
    // no save here
}

func recordRecentCoordinate(...) throws {
    try insertRecentCoordinate(...)
    try modelContext.save()
}

func recordBatchRun(...) throws {
    try insertRecentCoordinate(label: coordinateLabel, coordinate: coordinate, lastUsedAt: timestamp)
    modelContext.insert(BatchRunSummary(...))
    try pruneBatchRunSummaries()
    try modelContext.save()  // single atomic save
}
```

### WR-05: `#Unique` constraint on `RecentCoordinate` uses floating-point fields — hash collision risk

**File:** `GPSMetadataEditor/Features/BatchHistory/Models/RecentCoordinate.swift:6`

**Issue:**
```swift
#Unique<RecentCoordinate>([\.latitude, \.longitude])
```
`Double` is stored in SQLite as `REAL` (IEEE 754 64-bit). SwiftData's `#Unique` constraint maps to a `UNIQUE` index on those columns. Two coordinates entered as e.g. `52.5200080` vs `52.52000800000001` (a floating-point formatting difference) will not be recognised as duplicates, so the merge-on-upsert behaviour that the store relies on (setting `label` and `lastUsedAt` after insert) will silently create a second row instead of updating the existing one, growing the table unboundedly beyond `recentCoordinateLimit`.

This is inherent to using raw `Double` for a unique key. The store's `recordRecentCoordinate` prune path mitigates growth, but the intent (deduplicate same location) is broken.

**Fix:** Round coordinates to a fixed precision (e.g. 6 decimal places ≈ 0.11 m) before storing, so the stored values are always canonical:
```swift
init(label: String, latitude: Double, longitude: Double, lastUsedAt: Date) {
    self.label = label
    self.latitude = (latitude * 1_000_000).rounded() / 1_000_000
    self.longitude = (longitude * 1_000_000).rounded() / 1_000_000
    self.lastUsedAt = lastUsedAt
}
```
Alternatively store the coordinate as a rounded string key and use `#Unique` on that.

### WR-06: `MapStyleButton` has no VoiceOver `accessibilityLabel` beyond the button title

**File:** `GPSMetadataEditor/Features/CoordinateSelection/Views/MapStyleOverlay.swift:26-34`

**Issue:**
```swift
Button(style.label, systemImage: systemImage) {
    selectStyle(style)
}
.labelStyle(.iconOnly)
```
With `.labelStyle(.iconOnly)` the visible label is suppressed. VoiceOver will synthesise the button title from the `style.label` string argument, which is correct in principle. However, the selected state is conveyed only via `foregroundStyle` colour change — a purely visual cue. Screen-reader users have no way to discover which style is currently active.

**Fix:** Add `accessibilityAddTraits(.isSelected)` conditionally:
```swift
.accessibilityAddTraits(isSelected ? .isSelected : [])
```

### IN-04: `ProcessRunnerTests` has a single test and catches only `CocoaError` — other launch-failure types are flagged as unexpected

**File:** `GPSMetadataEditorTests/ProcessRunnerTests.swift:13-17`

**Issue:** The test asserts that launching a non-existent executable throws `CocoaError`. On macOS, `Process.run()` throws `CocoaError(.fileNoSuchFile)` for a missing executable, which this test correctly catches. However the comment says "expected — process launch failed with CocoaError" while the broader `catch` arm records any other error type as unexpected. If the implementation is ever changed to wrap the error differently (e.g. a custom `ProcessError`), this test silently passes the outer `do/catch` and records an unexpected failure via `Issue.record` rather than failing cleanly with a typed assertion.

The test has no assertion about the `CocoaError` code (`.fileNoSuchFile` vs other codes), so it accepts any `CocoaError` regardless of cause, which is weaker than intended.

**Fix:**
```swift
} catch let error as CocoaError where error.code == .fileNoSuchFile {
    // expected — missing executable
}
```
