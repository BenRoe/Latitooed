  <action>
    Three surgical changes to BatchHistoryStore.swift:

    CHANGE A — Replace recordRecentCoordinate (lines 34–60) with upsert pattern:

    The current implementation fetches (existingRecentCoordinate) then conditionally inserts.
    With #Unique declared on (latitude, longitude), SwiftData enforces uniqueness at the store
    level: an insert with a duplicate key merges into the existing record rather than creating
    a duplicate. The upsert pattern is: create a new instance, insert it, then set all mutable
    fields. If the (lat, lon) pair already exists, SwiftData updates the existing record's
    fields to the values set after insert.

    Replace the entire recordRecentCoordinate function body with:

        func recordRecentCoordinate(
            label: String,
            coordinate: CoordinateSelection,
            lastUsedAt: Date = Date()
        ) throws {
            let coord = RecentCoordinate(
                label: label,
                latitude: coordinate.latitude,
                longitude: coordinate.longitude,
                lastUsedAt: lastUsedAt
            )
            modelContext.insert(coord)
            // Set mutable fields after insert so SwiftData carries them on #Unique merge.
            coord.label = label
            coord.lastUsedAt = lastUsedAt
            try pruneRecentCoordinates()
            try modelContext.save()
        }

    CHANGE B — Remove existingRecentCoordinate(for:) private method (lines 123–127).
    The entire method is now dead code. Delete it completely.

    CHANGE C — Remove hasChanges guards in recordBatchRun (lines 99–101):

    The current form:
        if modelContext.hasChanges {
            try modelContext.save()
        }

    Replace with:
        try modelContext.save()

    Rationale (per DATA-LOW in RESEARCH.md Plan B): The swiftdata-pro skill rule states
    "There is no need to check modelContext.hasChanges before saving; just call save()
    directly." The guard is a no-op optimisation that adds noise and can mask the case
    where pruning caused deletions that need saving.

    Note: recordRecentCoordinate was already updated in CHANGE A to call save() directly.
    CHANGE C targets only the hasChanges guard remaining in recordBatchRun (the second
    occurrence at lines 99–101 of the original file).
  </action>
